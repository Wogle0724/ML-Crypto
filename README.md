# Crypto Predictor

A cryptocurrency price prediction platform built with a microservices architecture.
Prices are fetched hourly via Airflow DAGs, stored in MySQL, served through a Node.js REST API,
and predictions are generated by a Python FastAPI ML service backed by a PyTorch LSTM model.
All experiments are tracked in MLflow and surfaced in a React + lightweight-charts dashboard.

## Services

| Service      | URL                       | Notes                            |
|--------------|---------------------------|----------------------------------|
| Frontend     | http://localhost:3000     | React + Vite dev server          |
| Backend API  | http://localhost:4000     | Node.js + Express                |
| ML Service   | http://localhost:8000     | Python FastAPI + PyTorch         |
| Airflow      | http://localhost:8080     | Login: admin / admin             |
| MLflow       | http://localhost:5000     | Experiment & model registry UI   |
| MySQL        | localhost:3306            | DB: crypto_db                    |
| Redis        | localhost:6379            | Price + prediction cache         |
| PostgreSQL   | (internal only)           | Airflow metadata DB              |

## Getting Started

### Prerequisites

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) (v24+)
- Docker Compose v2 (`docker compose` — no hyphen)

### 1. Configure environment

The `.env` file is pre-populated with development defaults.
Before first run, generate a real Airflow Fernet key:

```bash
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

Paste the output into `.env` as `AIRFLOW_FERNET_KEY`, and set a random string for `AIRFLOW_SECRET_KEY`.

### 2. Start all services

```bash
docker compose up --build -d
```

Startup order (managed by `depends_on` + healthchecks):
1. MySQL, Redis, PostgreSQL start first
2. Backend + ML service start once MySQL and Redis are healthy
3. `airflow-init` runs once (migrates DB, creates admin user), then exits
4. Airflow webserver + scheduler start after `airflow-init` completes

> First build takes several minutes — PyTorch is large.

### 3. Run the data pipeline

After all services are healthy:

1. Open **http://localhost:8080** (admin / admin)
2. Trigger the `fetch_prices` DAG — fetches 90 days of hourly BTC/ETH data from CoinGecko and stores it in MySQL
3. Trigger the `train_model` DAG — trains the LSTM on the price history and registers it in MLflow
4. Verify the model appears at **http://localhost:5000** under `LSTMPriceModel / Production`

### 4. View the dashboard

Open **http://localhost:3000** — you will see:
- A blue price history line (most recent 2000 hourly closes from MySQL, ~83 days)
- A dashed red prediction overlay (3 hourly forward steps from the LSTM)

### 5. Test the APIs directly

```bash
# Most-recent BTC prices from MySQL (Redis-cached 60s)
curl http://localhost:4000/api/prices/bitcoin

# LSTM prediction via backend gateway (Redis-cached 5min)
curl http://localhost:4000/api/predict/bitcoin

# Call the ML service directly
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{"coin": "bitcoin"}'
```

### 6. Stop services

```bash
docker compose down
```

To also remove all volumes (MySQL data, MLflow artifacts, Airflow logs):

```bash
docker compose down -v
```

> **Between sessions:** always run `docker compose down` before shutting down your machine.
> Force-quitting Docker Desktop with containers running can corrupt the postgres volume,
> requiring `docker compose down -v` and a full pipeline re-run on next boot.

## Data & ML Flow

```
CoinGecko API (90 days, hourly)
    │  fetch_prices DAG (@hourly, schedule)
    ▼
MySQL  crypto_db.prices  (coin, ts, open, high, low, close, volume)
    │  train_model DAG (@daily, schedule)
    ▼
features.py → compute_features()
  12 columns: open, high, low, close, volume,
              RSI_14, MACD_12_26_9, MACDh_12_26_9, MACDs_12_26_9,
              BBL_5_2.0, BBM_5_2.0, BBU_5_2.0
    │  sliding windows of 60 timesteps, normalised by first close
    ▼
LSTMModel  (input 12×60 → hidden 64 × 2 layers → output 1)
  trained with MSE loss, Adam, 20 epochs
    │  val MSE (normalised) ~0.0005
    ▼
MLflow Model Registry  →  LSTMPriceModel / Production
    │  ml-service loads on each /predict request
    ▼
POST /predict  →  { coin, predictions: [{time, value}, ...] }
    │  backend caches 5 min in Redis
    ▼
Dashboard  →  PredictionOverlay  (dashed red line, 3 hourly steps)
```

## Key Implementation Notes

### Airflow metadata uses PostgreSQL, not MySQL
Airflow 2.10.5 migration `686269002441` uses `PREPARE/EXECUTE` syntax that MySQL
does not support. A dedicated `postgres:16-alpine` service is used for Airflow's
metadata database; MySQL stores only crypto price data.

### MLflow artifact serving
The MLflow server runs with `--serve-artifacts` and `--default-artifact-root mlflow-artifacts:/`.
Without this, remote clients (Airflow containers) receive a `file:///mlflow/...` artifact
URI and attempt to write directly to the MLflow container's filesystem, causing
`PermissionError`. With `--serve-artifacts`, uploads go through the MLflow HTTP API.

### Bollinger Bands computed manually
`df.ta.bbands(append=True)` in pandas-ta 0.4.71b0 silently returns `None` under some
numba JIT configurations, leaving the DataFrame without `BBL/BBM/BBU` columns.
`features.py` computes Bollinger Bands with `pandas.Series.rolling()` instead.

### SQLAlchemy 2.x + pandas 2.x compatibility
`pd.read_sql(engine, ...)` is removed in pandas 2.x. All database reads use the
`engine.connect()` → `conn.execute(text(...))` → `pd.DataFrame(result.fetchall())` pattern.

### LSTM pickle path in ml-service
The model is pickled in the Airflow container with `model.lstm.LSTMModel` as the class
path (via `sys.path.insert(0, "/opt/airflow/ml_service_app")`). The ml-service's
`WORKDIR` is `/app`, so `model.lstm` would not resolve without adding `/app/app` to
`sys.path`. `load.py` inserts its own grandparent directory at import time.

### MySQL duplicate rows
`fetch_prices` fetches 90 days of data on every hourly run without a `UNIQUE INDEX`
on `(coin, ts)`, so each run inserts duplicate timestamps. Read paths handle this:
- **Backend:** `SELECT DISTINCT ... ORDER BY ts DESC LIMIT 2000` (most recent ~83 days)
- **ML service:** `.drop_duplicates(subset=["ts"])` after loading from MySQL
